# Dog Office 2.0 技术改造清单

> 蓝图依据：2026-05 与老板敲定的 8 项设计决策（混合并行/契约先行、同目录纪律隔离、
> 待验收生命周期、例外管理通信、全家桶 9 角色、照单招人、老板终验收+交接档案、
> 多项目并行+标签切换）

## Phase 1：地基（数据层 + dog CLI）

**1.1 项目维度**
- [ ] `agents` 表加 `projectId`、`projectName`、`projectDir` 列（含迁移）
- [ ] 新建 `projects` 表：id、名称、目录、状态（进行中/待验收/已归档）、创建时间
- [ ] types / store / server 事件全链路带 projectId

**1.2 状态机扩展**
- [ ] AgentStatus 加 `submitted`（待验收）状态
- [ ] 工位视觉：待验收姿态（狗举手交卷？）+ 状态色

**1.3 dog CLI 新命令**
- [ ] `dog team` — 列出同项目在岗成员（查 server API）
- [ ] `dog send <狗名> "<消息>"` — 点对点消息（server 路由到目标终端，复用 sendToTerminal）
- [ ] `dog submit "<说明>"` — 声明交付，状态→submitted，通知质检/PM
- [ ] `dog hire <角色> --task "<任务>"` — 仅 PM 可用（校验调用者 role），招聘走 server → 开终端
- [ ] 所有子命令通过 `DOG_PROJECT_ID` 环境变量继承项目归属（同 parentId 机制）

**1.4 角色系统**
- [ ] 启动时按角色拼接 prompt：`_common.md` + `roles/<role>.md`
- [ ] 角色注册表：9 角色的 id、名称、默认工具、默认模型、emoji
- [ ] 每角色模型/工具可在招聘时覆盖（不写死）

**1.5 server API**
- [ ] `/api/team?projectId=` — 成员列表
- [ ] `/api/send` — 狗间消息路由
- [ ] `/api/hire` — PM 招聘入口（校验编制？v1 先不校验，靠 prompt 纪律 + 老板可见）
- [ ] 升级事件类型：`escalation`（质检上报 PM 时 dashboard 也收到，弹通知）

## Phase 2：界面

**2.1 项目标签栏**
- [ ] 顶部项目 tabs：进行中的项目 + "+ 新项目"按钮
- [ ] 办公室只渲染当前选中项目的狗；切 tab 切团队
- [ ] 项目级状态徽章（进行中 N 人在岗 / 待你验收 / 已归档）

**2.2 新项目流程**
- [ ] "+ 新项目" → 选项目文件夹（系统目录选择器）→ 填项目名 → 配 PM 的工具/模型 → 开工
- [ ] PM 终端 cwd = 项目文件夹

**2.3 招聘弹窗重做**
- [ ] 9 角色卡片选择（带 emoji、职责一句话）
- [ ] 每角色默认工具/模型预填、可改
- [ ] 手动招聘的狗也归属当前项目

**2.4 验收交互**
- [ ] 项目状态"待你验收"时：dashboard 顶部横幅 + 通知
- [ ] 验收面板：PM 的验收报告展示 + 「通过」/「打回（写意见）」两个按钮
- [ ] 打回意见直接发到 PM 终端

## Phase 3：体验完善（可延后）

- [ ] 升级事件的可视化（病历卡片展示在 PM 详情面板）
- [ ] 返工轮次显示在工位卡片（小角标：第 2 轮）
- [ ] HANDOVER.md 在归档项目详情里直接可读
- [ ] 团队解散动画（全员依次走出办公室）
- [ ] 重新打包 .dmg（带新图标）

## 风险与待验证

- `dog hire` 由 PM 的 Claude 在终端里调用 → 新终端窗口由 osascript 打开，
  PM 的会话会被新窗口抢焦点——需要验证体验是否可接受（或 hire 走后台静默开窗）
- 多项目并行时端口/通知的项目区分
- 质检"病历"依赖质检自觉记录——prompt 已写强制格式，实测看遵守率
